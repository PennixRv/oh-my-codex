# OMX leader mailbox notification chain analysis 2026-06-28

## Question
What is the current worker-to-leader notification path in `omx team`, and where does the synthetic `Read ... leader-fixed.json` reminder come from?

## Ranked synthesis

| Rank | Explanation | Confidence | Basis |
| --- | --- | --- | --- |
| 1 | Worker-to-leader delivery is mailbox-first. The worker does not directly inject prompt text into the leader pane as the primary transport. | High | Worker bootstrap, runtime `sendWorkerMessage`, mailbox queueing, and notify-hook dispatch logic agree. |
| 2 | The synthetic `Read ... leader-fixed.json` reminder text is generated by `buildLeaderMailboxTriggerDirective(...)` and may be persisted as a second mailbox message from `system` during fallback confirmation. | High | Direct code path from trigger directive builder to `notifyLeaderAsync` to `notifyLeaderMailboxAsync`. |
| 3 | Leader-visible nudging still exists as a separate layer: `team-leader-nudge` reads unread leader mailbox state and writes leader-attention / nudge state, which can surface user-visible orchestration reminders independently of the mailbox payload itself. | High | Direct code path in notify-hook leader nudge and leader attention state writer. |

## Evidence
- `src/team/worker-bootstrap.ts` instructs workers to ACK with `omx team api send-message ... to_worker:"leader-fixed"`.
- `src/team/runtime.ts` routes `toWorker === leader-fixed` through `sendLeaderMailboxMessage(...)`.
- `src/team/worker-bootstrap.ts` builds the leader trigger text as `Read .../mailbox/leader-fixed.json; ... Review it ...`.
- `src/team/mcp-comm.ts` persists the direct mailbox message first, then enqueues a dispatch request carrying the trigger text.
- `src/scripts/notify-hook/team-dispatch.ts` treats `leader-fixed` specially and returns `leader_mailbox_notified` without tmux `send-keys` injection.
- `src/team/runtime.ts` fallback path `notifyLeaderAsync(...)` persists a mailbox message from `system` containing the trigger text.
- `src/team/tmux-session.ts` `notifyLeaderMailboxAsync(...)` is explicitly documented as mailbox-only and calls `sendDirectMessage(teamName, fromWorker, leader-fixed, message, cwd)`.
- `src/scripts/notify-hook/team-leader-nudge.ts` separately reads `mailbox/leader-fixed.json`, computes unread count / action state, and writes leader attention state.

## Current chain
1. Worker starts and sends ACK or progress using `omx team api send-message` to `leader-fixed`.
2. Runtime persists the real worker payload into `.omx/state/team/<team>/mailbox/leader-fixed.json`.
3. Runtime also creates a dispatch request whose `trigger_message` is the synthetic `Read ... leader-fixed.json ...` instruction.
4. For `leader-fixed`, notify-hook dispatch no longer directly injects into the leader pane on the normal hook-drain path; it treats mailbox persistence as the notification boundary.
5. If fallback confirmation path runs, runtime may persist another mailbox message from `system` containing the trigger text. This is the synthetic reminder the user sees mixed into the same leader mailbox file.
6. Separately, leader nudge logic can synthesize visible leader reminders from unread mailbox state or idle-team state.

## Implication
The current fork mixes two different things in the same leader mailbox file:
- real worker semantic payloads
- control-plane reminder payloads generated by runtime / system logic

That mixing is why the mailbox can look like prompt injection even when direct tmux injection for leader dispatch has already been removed from the main dispatch path.

## Follow-up direction
The clean boundary is: keep worker payload in mailbox, keep reminder / attention state in dispatch or leader-attention state, and never persist synthetic system reminder text as another leader mailbox message.
